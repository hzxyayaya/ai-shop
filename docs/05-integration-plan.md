# AI 聊天式电商项目联调与冻结规则文档

## 1. 文档目的

本文档作为 `docs/05-integration-plan.md` 的联调与冻结规则补充版，用于统一管理 AI 聊天式电商项目在多 Agent 并行开发模式下的协作方式、变更流程、联调顺序和验收标准。

本文档目标：

- 明确联调总原则
- 强化“先冻结再开发”的执行规则
- 规范 Mock 使用与真实接口替换
- 规范字段与接口变更申请流程
- 明确各 Agent 之间的依赖关系
- 统一联调完成标准与冲突处理方式

本文档适用于以下 4 个并行开发角色：

1. 架构与契约 Agent
2. 后端业务 Agent
3. AI 模块 Agent
4. 前端 Agent

---

## 2. 联调总原则

### 2.1 文档优先于实现

联调过程中，统一以冻结文档为准：

- `docs/01-architecture-and-contract.md`
- `docs/02-database-design.md`
- `docs/03-api-spec.md`
- `docs/04-agent-task-board.md`
- `docs/03a-ai-chat-response-contract.md`

任何实现与文档不一致时，默认实现有问题，不默认文档过期。

### 2.2 契约优先于局部便利

- 不允许因为单模块开发方便而临时改接口
- 不允许因为局部实现卡住而擅自新增字段
- 不允许让前端、后端、AI 各自维护不同版本的契约

### 2.3 主链路优先于扩展能力

当前阶段优先保障以下主链路：

- 登录
- 商品浏览与搜索
- 加入购物车
- 立即购买
- 购物车勾选结算
- 订单查询
- 支付宝沙箱支付
- AI 搜索 / 推荐 / 加购 / 下单 / 查订单 / 支付引导

所有不影响主链路闭环的优化项都应让位于联调稳定性。

### 2.4 一轮联调只解决一层问题

每轮联调优先处理以下顺序：

1. 接口可达
2. 鉴权正确
3. 字段一致
4. 业务结果正确
5. 页面交互正确
6. 体验优化

禁止在一次问题处理中同时改数据库结构、API 契约、页面结构和 AI 输出协议。

## 3. 先冻结再开发的规则

### 3.1 冻结顺序

正式开发与联调必须遵循以下顺序：

1. 冻结架构边界
2. 冻结数据库结构
3. 冻结 API 契约
4. 冻结 AI 返回协议
5. 各 Agent 并行开发
6. 按顺序进入联调

### 3.2 未冻结不得进入正式联调

以下内容未冻结前，不允许进入对应模块正式联调：

- 商品字段未冻结，不进入商品联调
- `items_json` 未冻结，不进入订单联调
- 支付方式未冻结，不进入支付联调
- `ChatResponse` 未冻结，不进入聊天联调

### 3.3 冻结内容清单

以下内容当前阶段视为冻结项：

- 商品字段固定
- 页面范围固定
- JWT 鉴权固定
- 支付方式固定
- 订单模型固定为 `order_info + items_json`
- `ChatResponse` 顶层结构固定

## 4. 前端 Mock 的使用规则

### 4.1 Mock 使用目的

前端允许在后端真实接口未完成时使用 Mock，以提前开展：

- 页面布局开发
- 组件开发
- 状态管理开发
- 路由与页面流转开发
- ChatResponse 渲染开发

### 4.2 Mock 必须遵守的规则

- Mock 字段必须与 `docs/03-api-spec.md` 保持一致
- AI Chat Mock 必须与 `docs/03a-ai-chat-response-contract.md` 保持一致
- 不允许前端自造未冻结字段
- 不允许使用下划线字段替代对外驼峰字段
- 不允许把 Mock 数据结构当作正式契约反向要求后端适配

### 4.3 Mock 替换规则

前端替换真实接口时，必须逐模块替换，不允许整体混乱切换。

建议替换顺序：

1. `auth`
2. `product`
3. `cart`
4. `order`
5. `payment`
6. `chat`

### 4.4 Mock 退出条件

一个模块满足以下条件后，前端应停止继续使用 Mock：

- 对应真实接口已可访问
- 返回字段与契约一致
- 至少完成一轮主流程联调

## 5. 字段变更申请规则

### 5.1 哪些字段变更必须申请

以下变更必须先申请再执行：

- 新增字段
- 删除字段
- 字段重命名
- 字段类型语义变化
- 枚举值变化
- 驼峰 / 下划线输出规则变化

### 5.2 申请内容必须包含

字段变更申请至少需要说明：

- 变更字段名称
- 所属模块或表
- 当前状态
- 变更原因
- 对前端、后端、AI 的影响范围
- 是否影响历史 Mock、联调数据、数据库结构

### 5.3 审批顺序

字段变更必须按以下顺序处理：

1. 架构与契约 Agent 判断是否允许变更
2. 更新对应冻结文档
3. 后端与 AI 确认实现影响
4. 前端确认渲染与交互影响
5. 再进入代码修改

### 5.4 默认处理原则

- 非主链路字段，优先不改
- 主链路字段，优先兼容旧契约或延期到下一阶段
- 已进入联调的字段，原则上不在当轮联调中改动

---

## 6. 接口变更申请规则

### 6.1 哪些接口变更必须申请

以下情况必须走接口变更申请：

- URL 路径变更
- Method 变更
- 请求字段变更
- 响应字段变更
- 分页结构变更
- 鉴权要求变更
- 返回码语义变更

### 6.2 申请内容必须包含

接口变更申请至少需要说明：

- 接口名称与路径
- 变更前契约
- 变更后契约
- 变更原因
- 影响模块
- 是否影响已完成联调页面
- 是否需要兼容过渡

### 6.3 审批规则

- 所有接口变更以 `docs/03-api-spec.md` 为唯一更新入口
- `chat` 返回结构变更还需同步更新 `docs/03a-ai-chat-response-contract.md`
- 未更新文档前，不允许直接修改实现

### 6.4 紧急修正规则

如因阻塞主链路必须紧急调整：

1. 先记录问题
2. 先形成最小变更方案
3. 先补文档
4. 再同步修改相关模块

禁止先改后补。

---

## 7. 各 Agent 的依赖关系

## 7.1 架构与契约 Agent

职责：

- 维护冻结边界
- 审核字段与接口变更
- 协调跨模块冲突
- 决定是否允许进入下一轮联调

依赖：

- 依赖业务目标和当前冻结范围

被依赖：

- 被后端、AI、前端三个执行 Agent 共同依赖

## 7.2 后端业务 Agent

职责：

- 提供 `auth`、`product`、`cart`、`order`、`payment` 真实接口
- 保持返回结构与文档一致
- 维护真实业务状态与支付闭环

依赖：

- 依赖架构文档、数据库文档、API 文档

被依赖：

- 被前端 Agent 依赖
- 被 AI 模块 Agent 依赖

## 7.3 AI 模块 Agent

职责：

- 实现 `/api/chat` 编排
- 生成稳定 `ChatResponse`
- 调用后端业务能力完成加购、下单、查订单、支付引导

依赖：

- 依赖架构文档
- 依赖 API 文档
- 依赖 AI 返回协议文档
- 依赖后端业务接口已稳定

被依赖：

- 被前端聊天页依赖

## 7.4 前端 Agent

职责：

- 页面开发
- Mock 使用与替换
- 接口对接
- 页面交互与渲染

依赖：

- 依赖 API 文档
- 依赖 AI 返回协议文档
- 依赖后端与 AI 接口稳定

被依赖：

- 最终展示与验收依赖前端落地

---

## 8. 联调顺序建议

建议联调顺序固定为：

1. 联调前准备
2. `auth`
3. `product`
4. `cart`
5. `order`
6. `payment`
7. `chat`
8. 全链路回归

### 8.1 顺序原因

- `auth` 是所有用户态能力基础
- `product` 是购物、下单、AI 推荐的前提
- `cart` 依赖认证与商品
- `order` 依赖购物车与商品快照
- `payment` 依赖订单生成
- `chat` 依赖商品、购物车、订单、支付能力全部稳定

### 8.2 禁止逆序强行推进

以下行为禁止：

- 在 `order` 未稳定时主推支付联调
- 在基础业务未稳定时主推 AI 联调
- 让前端基于未冻结接口提前接真实联调

## 9. 每轮联调的完成标准

## 9.1 第一轮：认证联调完成标准

- 注册成功
- 用户名登录成功
- 邮箱登录成功
- JWT 保存成功
- 受保护接口可正确识别登录态
- 登录失效可正确返回 `401`

## 9.2 第二轮：商品联调完成标准

- 首页商品列表加载成功
- 搜索成功
- 商品卡片字段完整正确
- 商品弹窗浮层展示正确
- `imageUrl`、`shopName` 命名无偏差

## 9.3 第三轮：购物车联调完成标准

- 商品可加入购物车
- 同商品重复加购数量正确累加
- 购物车列表正确展示
- 数量修改成功
- 勾选状态更新成功
- 删除成功

## 9.4 第四轮：订单联调完成标准

- 立即购买可创建订单
- 购物车勾选结算可创建订单
- 订单列表可展示
- 订单详情可展示
- `items_json` 映射后的 `items` 结构正确
- 订单归属正确

## 9.5 第五轮：支付联调完成标准

- 创建支付成功
- 前端可跳转支付宝沙箱
- 支付回调成功
- `payment_record` 更新成功
- `order_info.pay_status` 更新成功
- 支付结果页展示正确

本地联调补充说明：

- 若后端运行在本机，`notify_url` 必须指向可从公网访问到本机 `8080` 的穿透地址
- 若前端运行在本机，支付发起页与结果页必须保持同一 origin
- 若异步回调存在延迟或本地 tunnel 不稳定，状态查询应允许主动向支付宝查单后再回写本地支付状态

## 9.6 第六轮：AI 联调完成标准

- `/api/chat` 稳定返回固定结构
- AI 搜索商品成功
- AI 推荐商品成功
- AI 加购成功
- AI 创建订单成功
- AI 查看订单成功
- AI 支付引导成功
- “第一个/第二个”引用解析正确
- 前端可稳定渲染文本、商品卡片、订单摘要和动作按钮

## 9.7 最终全链路回归标准

- 从 `/home` 可以完成浏览、加购、下单、支付
- 从 `/chat` 可以完成搜索、加购、下单、支付引导
- `/cart`、`/orders`、`/payment/result` 页面链路完整
- 不存在主链路字段漂移

## 10. 常见冲突处理规则

## 10.1 字段名冲突

处理顺序：

1. 以 `docs/03-api-spec.md` 和 `docs/03a-ai-chat-response-contract.md` 为准
2. 判断是实现错误还是文档缺失
3. 若确需调整，先走变更申请

### 典型冲突

- 后端返回 `image_url`，前端使用 `imageUrl`
- AI 返回 `shop_name`，前端只识别 `shopName`

默认结论：

- 对外统一使用驼峰，返回下划线视为实现不合规

## 10.2 状态枚举冲突

处理原则：

- 以冻结文档中的枚举为准
- 不允许单模块新增“临时状态”
- 如确需新增，必须补文档后统一改造

## 10.3 Mock 与真实接口冲突

处理原则：

- 先确认 Mock 是否符合冻结契约
- 若 Mock 错，前端修 Mock
- 若真实接口错，后端修接口
- 不允许为了兼容错误 Mock 而污染正式契约

## 10.4 AI 与后端职责冲突

处理原则：

- AI 负责编排，不负责直接写数据库
- 业务状态以后端真实模块为准
- AI 只通过工具调用或业务接口触发行为

## 10.5 页面交互与接口能力冲突

处理原则：

- 页面不能倒逼主契约越界
- 若页面需要新能力，先提交变更申请
- 若只是展示缺口，优先由前端在现有契约内调整交互

## 11. 每个 Agent 在提交结果时必须附带的联调检查项

## 11.1 架构与契约 Agent

提交时必须附带：

- 本次是否影响冻结文档
- 是否涉及字段、接口、枚举变化
- 是否已同步更新相关文档
- 是否允许进入下一轮联调
- 当前阻塞项列表

## 11.2 后端业务 Agent

提交时必须附带：

- 本次完成的接口清单
- 每个接口是否与 `docs/03-api-spec.md` 完全一致
- 是否需要 JWT
- 是否已覆盖错误场景和空数据场景
- 返回字段是否全部为对外驼峰
- 是否影响前端或 AI 模块现有联调

## 11.3 AI 模块 Agent

提交时必须附带：

- 本次支持的 intent 清单
- 本次支持的 action type 清单
- `ChatResponse` 是否完全符合协议
- 商品是否全部来自真实数据库结果
- 是否存在绕过后端业务模块的行为
- “第一个/第二个”引用解析是否已验证

## 11.4 前端 Agent

提交时必须附带：

- 当前使用 Mock 还是真实接口
- 已替换的真实接口清单
- 页面字段映射是否与契约一致
- 错误态、空态、登录失效态是否验证
- 动作按钮行为是否与 `action type` 对齐
- 当前仍受阻的接口或字段问题

---

## 12. 推荐交付节奏

建议采用如下协作节奏：

### 每日开始前

- 更新任务看板
- 确认当天联调模块
- 确认冻结文档无新增冲突

### 每个模块联调前

- 后端或 AI 提供可验证接口
- 前端确认 Mock 与正式契约一致
- 架构 Agent 确认无未决变更申请

### 每个模块联调后

- 记录已完成检查项
- 记录问题清单
- 记录是否允许进入下一轮
