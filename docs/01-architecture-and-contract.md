# AI 聊天式电商项目架构与模块边界文档

## 1. 文档目的

本文档作为 `docs/01-architecture-and-contract.md` 第一版初稿，用于冻结当前阶段 AI 聊天式电商项目的工程目录、模块职责、协作边界与并行开发约束。

本文档面向以下目标：

- 为前后端分离开发提供统一工程骨架
- 为前端、后端、AI 三方并行开发提供稳定边界
- 为数据库、接口、支付、聊天协议提供冻结依据
- 避免在当前阶段引入超范围设计

当前阶段目标是交付一个可开发、可联调、可演示、可闭环下单支付的工程化版本，而不是完整平台化电商系统。

---

## 2. 项目范围与约束

### 2.1 已确认范围

- 前后端分离
- 前端：Vue3 PC Web
- 后端：Spring Boot + WebFlux
- 数据库：PostgreSQL
- AI 模型：DeepSeek
- 商品形态：仅支持 SPU
- 用户体系：用户名/邮箱 + 密码
- 鉴权方式：JWT Bearer Token
- 支付方式：支付宝沙箱 PC 网页跳转支付
- 支持真实购物车
- 支持两种下单方式：
  - 立即购买
  - 购物车勾选后结算
- 订单模型：`order_info` 主表 + `items_json` 商品快照
- 首页风格：小红书式瀑布流
- 商品交互：点击后弹窗浮层聚焦展示，背景模糊
- AI 聊天页风格：ChatGPT / Gemini

### 2.2 明确不进入当前阶段

- SKU
- 库存
- 多规格
- 多商户
- 地址体系
- 优惠券
- 复杂营销
- 复杂订单拆表
- 发货履约
- 售后退款
- 小程序 / App / 多端统一架构

## 4. 各目录职责说明

### 4.1 `docs/`

职责：

- 沉淀项目范围、架构、接口、数据库、联调计划
- 作为多 Agent 并行开发的冻结依据
- 所有主流程变更先改文档，再改实现

### 4.2 `frontend/`

职责：

- 承载 Vue3 PC Web 前端工程
- 实现首页、聊天页、购物车、订单、登录注册、支付结果页
- 对接后端 REST / Chat 接口
- 不承载业务真相，只做状态呈现与交互编排

### 4.3 `backend/`

职责：

- 承载 Spring Boot + WebFlux 核心业务系统
- 管理用户、商品、购物车、订单、支付、聊天编排
- 对数据库、支付网关、AI 能力提供统一后端入口
- 作为唯一业务事实来源

### 4.4 `ai/`

职责：

- 管理 AI 提示词、输出协议、评测样例、检索策略说明
- 承载 DeepSeek 集成相关的协议文档、Prompt 模板、结构约束
- 不直接替代后端业务模块，不单独成为交易系统事实源

说明：

- 当前建议将 AI 的运行能力并入 `backend/chat` 模块实现
- `ai/` 目录主要用于沉淀提示词、协议、评测和实验材料

### 4.5 `sql/`

职责：

- 管理 PostgreSQL 表结构、迁移脚本、种子数据、校验脚本
- 冻结主表结构与索引约束
- 为本地、测试、联调环境提供统一数据库基线

### 4.6 `crawler/`

职责：

- 承载商品采集、清洗、标准化、入库前处理
- 输出可导入 `product` 表的数据
- 该目录不直接参与主交易链路

说明：

- 当前阶段可存在，但不进入主线交付路径
- 不允许 crawler 擅自扩展主商品字段

### 4.7 `monitor/`

职责：

- 存放监控配置、日志分析脚本、健康检查与告警规则
- 支撑部署后的可观测性

说明：

- 当前阶段以基础健康检查和日志观察为主
- 不应反向驱动主业务模型变更

## 5. 模块边界说明

## 5.1 前端负责什么

前端负责用户可见的展示层与交互层，包括：

- 登录、注册、登出、登录态保存
- 首页瀑布流展示与搜索交互
- 商品卡片展示与商品聚焦弹窗
- AI 聊天页消息展示、输入、动作按钮触发
- 购物车勾选、数量调整、删除、去结算
- 订单列表展示、支付状态展示、继续支付入口
- 支付跳转前后的页面承接
- 调用后端接口并按冻结契约渲染结果

前端不负责：

- 自行计算订单真值
- 自行判断支付成功
- 直接访问数据库
- 直接调用 DeepSeek 绕过后端
- 擅自定义与后端不一致的商品、订单、聊天协议

## 5.2 后端负责什么

后端负责业务编排、数据持久化与外部能力接入，包括：

- 用户注册、登录、JWT 生成与鉴权
- 商品列表、搜索、轻量详情查询
- 购物车真实写入与查询
- 立即购买与购物车勾选结算
- 订单生成、订单状态流转、商品快照固化
- 支付宝沙箱下单、回调验签、支付状态更新
- Chat 接口统一入口
- AI 工具调用编排与上下文管理
- PostgreSQL 数据存储与事务边界控制

后端不负责：

- 承担前端视觉表现
- 输出未冻结的临时字段作为主契约
- 绕过订单与支付模块直接修改状态
- 把 AI 输出当作数据库事实直接落库

## 5.3 AI 模块负责什么

AI 模块负责导购理解与响应编排，包括：

- 用户自然语言意图识别
- 商品检索编排
- 商品推荐话术生成
- “第一个/第二个”之类上下文引用解析
- 将用户请求转换为后端工具调用意图
- 生成固定结构的 ChatResponse

AI 模块不负责：

- 直接执行数据库写操作
- 直接生成虚构商品
- 绕过后端鉴权访问用户数据
- 擅自新增主流程字段、意图、动作协议
- 自行定义支付成功、订单成功等业务最终状态

## 6. 当前阶段必须冻结的内容

## 6.1 商品字段冻结

商品表名固定为 `product`。

商品字段固定为：

- `id`
- `category`
- `title`
- `price`
- `sales`
- `image_url`
- `shop_name`
- `embedding vector(2048)`

冻结规则：

- 当前只表示 SPU
- 不允许新增 SKU、库存、多规格相关字段进入主流程
- 不允许前后端使用不同商品字段集合

## 6.2 页面范围冻结

当前阶段页面范围固定为：

- `/login`
- `/register`
- `/home`
- `/chat`
- `/cart`
- `/orders`
- `/payment/result`

冻结规则：

- 可以做页面内部优化
- 不新增主流程页面作为联调依赖
- 不引入独立商品详情页作为当前主链路前置条件

## 6.3 订单模型冻结

订单采用简化模型：

- 主表：`order_info`
- 商品快照字段：`items_json`

冻结规则：

- 当前不拆 `order_item` 作为主实现
- `items_json` 必须可同时承载立即购买与购物车结算两类订单
- 不引入履约、发货、退款、售后等复杂状态链

## 6.4 支付方式冻结

支付方式固定为：

- 支付宝沙箱 PC 网页跳转支付

冻结规则：

- 当前不支持二维码支付
- 当前不支持 App 支付
- 当前不支持小程序支付
- 前端只负责跳转，支付结果以服务端回调和订单状态为准

## 6.5 鉴权方式冻结

鉴权方式固定为：

- JWT Bearer Token

请求头固定为：

```text
Authorization: Bearer <token>
```

冻结规则：

- 不切换为 Session/Cookie 作为主方案
- 不允许前端私自定义另一套 token 传递方式

## 6.6 AI 返回结构冻结

AI 聊天接口返回结构必须包含以下顶级字段：

- `intent`
- `message`
- `products`
- `orders`
- `actions`

冻结规则：

- 前端渲染只依赖该固定结构
- 后端 chat 模块与 AI 提示词都必须围绕该结构输出
- 不允许删除上述字段

建议约束：

- `products` 只返回真实商品结果
- `orders` 只返回当前用户可见订单摘要
- `actions` 只返回前端已定义可执行动作

## 7. 推荐模块划分

### 7.1 前端模块建议

- `app`：应用初始化、全局布局、全局状态挂载
- `views`：页面级路由视图
- `components`：可复用 UI 组件
- `services`：HTTP 请求封装与 API 调用
- `stores`：Pinia 状态管理
- `types`：前端契约类型
- `utils`：通用工具函数

### 7.2 后端模块建议

- `controller`：统一放置 HTTP 接口入口，如 `AuthController`、`CartController`、`OrderController`
- `service`：统一放置业务编排服务，如 `AuthService`、`CartService`、`OrderService`
- `dto`：统一放置请求和响应对象；当单一业务对象较多时，允许继续按 `auth/`、`cart/`、`order/`、`payment/`、`product/`、`chat/` 建子目录
- `entity`：统一放置数据库实体；当实体数量增加时，允许按业务建子目录
- `repository`：统一放置 Repository 与自定义 Repository 实现；当文件较多时，允许按业务建子目录
- `security`：统一放置 JWT、认证过滤器等安全相关实现
- `client`：统一放置第三方客户端，如支付宝网关客户端
- `model`：统一放置非 DTO、非 Entity 的领域内模型
- `monitor`：健康检查、日志、运行态观察
- `common`：统一返回、异常、常量
- `config`：配置类
- `infra`：数据库、外部客户端、基础设施适配

### 7.3 AI 子模块建议

- `prompts/`：系统提示词、业务 Prompt 模板
- `schemas/`：ChatResponse、意图、动作协议说明
- `evaluators/`：Prompt 与结果质量评测样本
- `mock-data/`：联调期间可用的 AI Mock 响应样例
- `docs/`：AI 策略、约束、测试记录

## 8. 当前阶段允许扩展但不进入主线的内容

以下内容允许保留为后续扩展方向，但当前不进入主线实现：

- 首页推荐排序策略优化
- 商品分类筛选增强
- 聊天接口从普通 JSON 升级为 SSE 流式输出
- AI 个性化推荐
- 更精细的向量召回与重排
- 支付状态轮询优化
- 独立商品详情页
- 订单详情页
- 基础埋点与行为分析
- 监控大盘可视化
- 爬虫数据清洗规则增强

规则：

- 扩展项只能放入建议区、预研区、支线目录
- 不得作为当前主链路联调前置依赖

## 9. 并行开发时禁止擅自修改的内容

以下内容属于跨 Agent 冻结边界，任何 Agent 不得单独修改：

- `product` 表名
- 商品固定字段集合
- `embedding vector(2048)` 约束
- 前端页面固定范围
- `order_info + items_json` 的订单主模型
- JWT Bearer Token 鉴权方案
- 支付宝沙箱 PC 网页跳转支付方案
- ChatResponse 顶级字段结构
- 当前只支持 SPU 的商品定义
- “立即购买”和“购物车勾选结算”两种下单方式

以下内容必须先改文档，再改代码：

- 数据库主表结构
- API 主响应字段
- 订单状态枚举
- 支付状态枚举
- AI intent 枚举
- 前端主路由范围
- 后端目录分层方式

以下内容允许各 Agent 在各自边界内自主调整：

- 目录内文件组织微调
- 组件命名细化
- Service / DTO / VO 具体命名
- 内部实现细节
- 样式实现方式
- 非主契约级别的日志、监控、测试代码

## 10. 并行协作规则

### 10.1 文档优先

任何主流程修改顺序必须为：

1. 先更新冻结文档
2. 再更新数据库 / API 文档
3. 再进入实现开发

### 10.2 契约优先

- 前端先以冻结接口契约开发
- 后端按契约提供真实接口
- AI 输出必须满足前端渲染契约

### 10.3 禁止越层

- 前端不能直接访问数据库或模型服务
- AI 不能直接写业务库
- 爬虫不能直接改线上交易表结构
- 支付状态不能以前端页面结果作为最终真相

## 11. 当前阶段交付标准

当以下能力全部跑通时，视为当前阶段达成目标：

- 用户可注册、登录，并维持 JWT 登录态
- 首页可展示商品瀑布流
- 商品搜索可用
- 商品弹窗浮层可正常展示与操作
- 用户可真实加入购物车并修改数量
- 用户可立即购买
- 用户可从购物车勾选后结算
- 订单可生成并可查询
- 支付宝沙箱支付链路可跑通
- AI 可搜索商品、推荐商品、触发加购与购买引导
- AI 返回结构可被前端稳定渲染
